# Agent目前最大的瓶颈是什么？

## 核心论点

Agent 被拆成了两个截然不同的世界：

### 干净世界（Claude Code / Hermes / OpenClaw / Coding Agent）
- 任务所有者 = 执行者，即时验收、即时纠正
- 环境反馈清晰（编译/测试用例）
- 可回滚（git revert）
- 所有博客/论文/框架都在卷这个

### 脏世界（企业级 Agent）
- 你做的不是 Agent，是「脏世界 → 干净世界」的翻译
- 数据是脏的（扫描件、文件名混乱、版本矛盾）
- 业务逻辑是模糊的（"合理期限"、"明显"等词定义不清）
- 错误不可逆（漏审可能赔钱）
- 用户既要又要（快、准、便宜、简单）

## 五堵墙（Five Walls）

### 第一堵墙：数据是脏的
30% 是扫描件，15% 是 Word 里嵌着截图的扫描件，5% 是图片格式的扫描件再被人截了图。文件名互相矛盾，"最终版"有 4 个，v2 比 v3 还新。光是搞清楚哪个文件是当前有效版本，就要先做半个月的数据治理。

### 第二堵墙：半结构化数据
完全结构化 → SQL Agent；完全非结构化 → LLM 反而不难。最难的是中间层：有格式但格式不严格，有结构但结构会破。表格某行突然变成合并单元格，条款编号编到 3.2.1 之后突然冒出"3.2.1.附"，正则解不动，LLM 看不全。

### 第三堵墙：长上下文是伪命题
厂商说支持 100 万 token，但真实做复杂推理的有效窗口可能只有 8K-32K。找事实（needle-in-a-haystack）和推理多个事实之间的关系完全是两种任务。企业文档互相引用、互相依赖但没有编译器，不能用线性 ReAct 处理。

### 第四堵墙：业务逻辑本身就是模糊的
"合理期限"、"明显的安全标志"、"必要时"、"重大不利影响"——法律条文必须留解释空间。企业 Agent 不能假装规则清晰，因为规则本身就不清晰。这一点 Coding Agent 永远不会遇到。

### 第五堵墙：用户既要又要还要
要准、要快、要便宜、要简单。四个互相矛盾的需求之间找一条能走的路。

## 五层解法（Five-Layer Solution）

### 第一层：脏数据预处理管道
让 LLM 远离原始数据。直接喂多模态 LLM 有三个硬伤：
1. **黑盒**：拿不到中间产物，出事无法回溯
2. **不可复现**：温度设 0 也不是完全确定的
3. **数据不是只给 Agent 用的**：做离线预处理管道，产出结构化数据资产

正确做法：文档解析（专业 OCR + LLM 互补）→ 结构化索引（章节树/图表位置/引用关系图谱）→ 分级存储（原文/摘要/索引，LLM 不直接读原文）

这一层跟 LLM 没关系，是数据治理。80% 的失败项目死在这一层。

### 第二层：混合解析
核心问题：哪些字段交给规则、哪些交给 LLM。

- **结构化字段**（金额/日期/编号）→ 正则 + 解析器 → 100% 确定，零 token 消耗，可审计可回滚
- **模糊文本**（自由表述/隐式引用/需要推理的条款）→ LLM，但只给相关片段
- 两路结果在索引层合并

**核心原则**："LLM 是用来吸收不确定性的，不是用来制造不确定性的。"

### 第三层：长上下文治理
必须分治：
1. 按章节/主题/条款拆成独立片段，每个片段控制在有效推理窗口（8K-32K）
2. 每个片段配上需要的上下文（引用关系 → 图而不是树）
3. 推理结果在顶层汇总，不在底层交叉

Coding Agent 的线性 ReAct 循环在这里行不通，必须做图遍历：先建引用图，再按依赖顺序审查，最后汇总。

### 第四层：模糊逻辑的规则化
新手的错误：把模糊词原封不动塞给 LLM → 不一致、不可解释、不可协商。

正确做法：
1. 对每个模糊条款，离线用 LLM 生成 3-5 个具体化的判断标准
2. 这些标准和业务方一起 review 确认
3. Agent 不判断"是否合理"，只判断是否满足显式标准
4. 标准本身配置化、可版本化、可追溯

**核心原则**："让 LLM 做判断，不让 LLM 做定义。"

### 第五层：用户预期管理
不可能三角(快/准/便宜)让用户自己选：
- **快速模式**：粗筛 + 关键词 + 规则引擎，不调 LLM，秒级返回，覆盖 80%
- **标准模式**：LLM 逐条审查，分钟级返回，覆盖 95%
- **深度模式**：多模型交叉验证 + 强制人工复核，小时级返回，覆盖 99%

三种模式共用同一套预处理管道、同一套索引、同一套规则化标准，区别只在调度策略。

## 评估方式（Evaluation）

Coding Agent 有 test（SWE-Bench），企业 Agent 没有标准答案。正确做法：

1. **和基线比**：比没有 Agent 的时候好就是有效果。不需要完美，只需要比昨天好
2. **量置信度**：好的 Agent 是高风险揪出 80% 且不误报，困难部分标"建议人工复核"
3. **做盲测**：用人工审查过的历史案例做双盲对比，不一致处请第三方裁决
4. **监控漂移**：法规变/业务规则变/数据分布变，Agent 能力会漂移，需持续评估

**核心哲学**：追求相对改进，不追求绝对数字。在脏世界里，相对改进就是一切。

## 关键引用

> "你学的Agent技术，和你实际做的Agent，根本不是一件事。"

> "Claude Code 和 Hermes 做的是干净世界 → 干净世界的翻译；你做的是脏世界 → 干净世界的翻译。"

> "LLM 是用来吸收不确定性的，不是用来制造不确定性的。"

> "让 LLM 做判断，不让 LLM 做定义。"

> "你卖的不是 Agent，是确定性产品。"

> "没有一招必杀。Coding Agent 的世界里确实有，因为那个世界是干净的。你的世界不是。"
